# DEC-004 — GACCard as a First-Class Backend Primitive

> **Status:** Locked 2026-04-12. Operationalizes `[[canon/specs/BLUEPRINT-PLANNER]]` §Data Models by committing to build the GACCard entity as a real backend service, not a markdown convention.

## The Decision

The **GAC queue is no longer just hand-edited markdown** under `intents/GAC-NNN/`. It is a first-class backend primitive with:

1. **Zod schema** in `shared/schemas/gac-card.ts` matching `[[canon/specs/BLUEPRINT-PLANNER]]` §Data Models verbatim (id, category, priority, question, options[], answer, result_action, connections, context).
2. **SQLite entity** in the services DB via Drizzle, indexed on `status`, `category`, `priority`, `created_at`.
3. **State machine** enforced server-side: `pending → answered | deferred | promoted`. Transitions are append-only; history is queryable.
4. **Two-layer filesystem mirror** at `.superman/gac/<NNN>/` matching the Intent two-layer pattern (Appendix A.6 of `[[_meta/SELF-POLLINATION-FINDINGS]]`): `card.md` is the human-readable source of truth, `status.json` is the runtime index mirror. The chokidar-based watcher keeps the DB in sync with the filesystem.
5. **HTTP routes** under `services/core/blueprint/routes.ts`: `GET /gac`, `GET /gac/:id`, `POST /gac`, `POST /gac/:id/answer`, `POST /gac/:id/defer`, `POST /gac/:id/promote`.
6. **MCP tools** exposed to agents: `gac_list`, `gac_show`, `gac_answer`, `gac_defer`, `gac_promote`. Follows the `Ema.MCP` tool registry pattern (Appendix A.2).
7. **Renderer binding** via `apps/renderer/src/components/blueprint/` consuming the `@ema/shared` types. The Blueprint vApp becomes the default UI for the GAC queue.

## Why

### The pattern we've been running manually

The session that produced this decision *was itself the Blueprint Planner operated manually*. The human was reviewing GAC cards, picking options, and the system was recording decisions into canon. `[[canon/specs/BLUEPRINT-PLANNER]]` predicted exactly this — the bootstrap v0.1 process is "the Blueprint Planner run by a human through a Claude.ai conversation." This decision promotes that process from "manual markdown editing" to "a service that runs the loop."

### The two-layer symmetry

EMA's Intent system (Appendix A.6) already has the two-layer pattern: filesystem source of truth (`.superman/intents/<slug>/intent.md` + `status.json`) + SQLite index for queryability. GACCards are the same primitive for a different lifecycle (`question → options → resolution`) as Intents are for (`problem → work → done`). Reusing the pattern keeps the architecture consistent.

### Unblocking the queue

Hand-editing 10 GAC card markdown files over a session is fine. Hand-editing 100 over a year is not. Promoting GACCard to a service means: new cards can be auto-generated by agents during research passes, answered cards can trigger `result_action: create_canon` automatically, and the queue becomes an observable backlog instead of a folder that decays.

## What This Changes

| Area | Before | After |
|---|---|---|
| `intents/GAC-NNN/README.md` | Sole source of truth, hand-edited | Still the source of truth in the filesystem layer; `card.md` file rename happens during Phase 2 port |
| `services/core/blueprint/` | Does not exist | New service with schema, state machine, routes, MCP tools, filesystem watcher |
| `shared/schemas/` | No `gac-card.ts` | New Zod schema matching BLUEPRINT-PLANNER §Data Models |
| `apps/renderer/src/components/blueprint/` | Not implemented | New vApp consuming `@ema/shared` types |
| Agent loop | No way for an agent to log a gap | `gac_create` MCP tool lets any agent raise a card |
