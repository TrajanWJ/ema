Enhancement pass: canonical/derived labels, identity/link separation, lineage spine, grounded bootstrap

1. Canonical vs Derived Stores

CANONICAL (source of truth)
intents table in SQLite — the graph
intent_edges table — relationships
intent_events table — lineage log
All mutations go through Ema.Intents context module. No other module writes directly.
DERIVED (projections, regenerable)
.superman/intents/<slug>/ — disk scratchpad
vault/intents/ — markdown notes for wiki
vault/system/state/intents.md — SystemBrain snapshot
ema intent tree CLI output — rendered view
All regenerable from canonical store. Loss = rebuild, not data loss.
Three Core Truths — coequal, not intent-monist
Semantic truth → intents, intent_edges, intent_events
Operational truth → executions, sessions, proposals, tasks, goals in their home domains
Knowledge truth → curated vault/wiki/docs/indexed project memory
intent_links bridges semantic truth to operational truth. Context assembly bridges semantic truth to knowledge truth. Projections are downstream of these truths, not peers to them.
This keeps the system disciplined: the intent graph names what the system is trying to do, operational tables record what actually happened, and curated knowledge stores record what is worth remembering.

2. Durable Identity vs Transient Links

intents table — separated into durable core + mutable state + transient links
── DURABLE IDENTITY (rarely changes after creation) ── id string "int_<ts>_<rand>" title string required slug string unique, auto-from-title description text optional level integer 0-5 (vision→execution) kind string goal|question|task|exploration|fix|audit|system parent_id FK → intents project_id FK → projects source_fingerprint string unique (dedup key) source_type string brain_dump|proposal|execution|harvest|goal|structural|crystallized|manual inserted_at datetime ── MUTABLE STATE (changes as work progresses) ── status string planned|active|researched|outlined|implementing|complete|blocked|archived phase integer 1-5 (research→harvest) completion_pct integer 0-100 (propagated from children) clarity float 0.0-1.0 energy float 0.0-1.0 priority integer 0-4 (P0=immediate → P4=someday) autonomy_level string full|supervised|gated fitness_score float 0.0-1.0 (execution success rate) confidence float 0.0-1.0 (how trusted this intent structure is) provenance_class string high|medium|low confirmed_at datetime tags text JSON array updated_at datetime ── TRANSIENT LINKS (join table: intent_links) ── Moved OUT of intents table. Many-to-many via intent_links: intent_links { intent_id FK → intents linkable_type string "execution"|"proposal"|"task"|"goal"|"brain_dump"|"session"|"harvest"|"vault_note"|"doc" linkable_id string the foreign ID role string "origin"|"evidence"|"derived"|"related"|"superseded"|"context" provenance string "manual"|"approved"|"execution"|"harvest"|"cluster"|"import" inserted_at datetime } WHY: An intent can have multiple executions over time, multiple sessions contributing, a proposal AND a brain dump as co-origins, and several knowledge artifacts supporting it. Fixed FK columns can't express this.

3. Lineage Spine (intent_events)

intent_events — append-only log answering "where did this come from? what happened?"
intent_events { id string auto intent_id FK → intents event_type string created|status_changed|phase_advanced|linked| unlinked|reparented|merged|split|archived| execution_started|execution_completed| confirmed|promoted|demoted|crystallized| outcome_recorded|projection_rebuilt payload text JSON — what changed (old→new status, linked ID, etc.) actor string "system"|"user"|"agent:<slug>"|"harvest"|"cluster" inserted_at datetime } EXAMPLE LINEAGE for a brain dump → executed intent: 1. created {source: "brain_dump", brain_dump_id: "bd_123"} 2. linked {type: "execution", id: "exec_456", role: "derived"} 3. status_changed {old: "planned", new: "active"} 4. phase_advanced {old: 1, new: 2, execution_id: "exec_456"} 5. execution_completed {execution_id: "exec_456", outcome: "success"} 6. outcome_recorded {what_worked: ["..."]} 7. status_changed {old: "active", new: "researched"} NOT full event sourcing — state lives in intents table. This is an audit trail. Read it to understand provenance. Delete old events freely (they're not needed for correctness).
Provenance / confidence / confirmation rules
High confidence → manual creation, approved proposals, execution-backed signals. These can create intents directly.
Medium confidence → brain dumps, harvested sessions, vault seeding. These can create leaf intents, usually with provenance_class=medium.
Lower confidence until confirmed → clustering, structural auto-analysis, crystallization inference. These should create candidates, suggested edges, or low-confidence intents first.
Lightweight confirmation thresholds: repeated occurrence, multi-source corroboration, operator confirmation, or successful execution/outcome linkage.
Recursive/reflexive creation is allowed, but promotion to execution-ready or curated knowledge requires evidence. The graph may speculate; the curated wiki should not.

Bootstrap safety rails — make recursive usefulness safe
Generated goes to generated spaces first → .superman/intents/, generated state notes, harvested notes, scratchpads.
Curated is protected → host-machine curated docs and curated EMA wiki pages are not overwritten by projections.
Imported material keeps provenance → every note/link knows whether it came from host_machine, agent_vm, repo docs, harvest, or system generation.
No blind merge → semantically conflicting sources stay separate until operator confirmation or a promotion rule resolves them.
Promotion into curated wiki truth should require confidence thresholds, corroboration, or operator approval. Scratch material from agent VM remains provisional unless promoted.
Execution-ready context assembly — what gets included, and why
Always include → the intent itself, parent/child spine, direct dependencies, linked executions/proposals/tasks, and highest-confidence lineage events.
Prefer next → curated EMA wiki pages, project-local specs/notes under ~/.local/share/ema/vault/projects/ema/, repo docs, and directly linked host-machine docs.
Include conditionally → recent session captures, harvested notes, execution scratchpads, and imported archives only when linked, recent, or corroborating.
Avoid by default → noisy archives, stale generated projections, unconfirmed cluster output, and broad vault search dumps without project relevance.
Ranking rules: provenance-aware inclusion, bounded context budgets, curated over noisy, local project docs over general archives, and direct intent linkage over semantic similarity.
Bootstrap implementation should extend existing Ema.Memory.ContextAssembler and Ema.Intelligence.ContextBuilder, not invent a second assembler.
