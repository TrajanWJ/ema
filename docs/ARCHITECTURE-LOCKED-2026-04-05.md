# EMA — Locked Architecture (2026-04-05)

This document is the **permanent spine** of EMA. Everything grows around what is defined here. A coding agent can implement directly from this document.

---

## 1. Final Core Ontology (LOCKED)

Six canonical objects. No others enter the core loop.

### Project

```
DB: projects table (exists)
ID: "proj_" prefix, string PK
```

| Field | Type | Purpose |
|-------|------|---------|
| slug | string (unique) | Human-readable identifier, used as filesystem key |
| name | string | Display name |
| status | enum | incubating → active → paused → completed → archived |
| linked_path | string | Filesystem path to project root |
| github_repo_url | string | Optional remote |
| settings | map | Per-project config (approval policy, agent defaults) |

**Owns:** Tasks, Proposals, Executions (via project_slug), Seeds, Responsibilities, Agents  
**Filesystem:** `{linked_path}/.superman/` contains intents, context, project memory  
**Role in loop:** Scoping container. Every Execution belongs to exactly one Project.

---

### Intent

```
Storage: FILESYSTEM ONLY — .superman/intents/{slug}/
DB: NOT a table. Represented by execution.intent_slug (string reference)
```

**Directory structure (already implemented in IntentFolder):**
```
.superman/intents/{slug}/
  ├── intent.md          # What this intent is about
  ├── result.md          # Latest execution result
  ├── execution-log.md   # Append-only log of all executions
  └── status.json        # Machine-readable state
```

| Concept | Implementation |
|---------|---------------|
| Intent creation | `IntentFolder.create(project_path, slug, content)` |
| Intent status | `IntentFolder.read_status(project_path, slug)` |
| Result writing | `IntentFolder.write_result(project_path, slug, result)` |
| Log appending | `IntentFolder.append_log(project_path, slug, exec_id, mode, result)` |

**Intent is NOT a DB entity.** It is a filesystem-resident knowledge accumulator. Executions reference it by slug. This is final.

**Why filesystem:** Intents are semantic memory — they grow, accumulate context, and get read by agents as working documents. DB rows can't serve this purpose. The slug is the join key between DB executions and filesystem state.

**Future hook point (intent clustering):** A `BrainDump.Clusterer` module can group unprocessed brain dump items by embedding similarity, create intent folders, and generate seeds — all without touching the Execution table or core loop. This is purely additive.

### Proposal

```
DB: proposals table (exists)
ID: "prop_" prefix, string PK
```

| Field | Type | Purpose |
|-------|------|---------|
| title | string | What this proposes |
| summary | string | One-paragraph description |
| body | string | Full proposal content |
| status | enum | generating → queued → reviewing → approved / redirected / killed / failed |
| confidence | float | 0.0-1.0, set by Debater |
| estimated_scope | enum | xs / s / m / l / xl |
| project_id | FK → Project | Scoping |
| seed_id | FK → Seed | Origin (nullable — manual proposals have no seed) |
| source_fingerprint | string | Dedup key |

**Lifecycle:** Generated by ProposalEngine OR created manually via API. Pipeline stages (generate → refine → debate → tag) are internal to ProposalEngine and do not affect the core loop. The core loop sees proposals only after they reach status "queued."

**Terminal states:** approved (→ creates Execution), redirected (→ creates 3 new Seeds), killed (→ logged to KillMemory).

### Execution

```
DB: executions table (exists)
ID: "exec_" prefix, string PK
```

**This is the central object of EMA.** Every meaningful action becomes an Execution.

| Field | Type | Purpose |
|-------|------|---------|
| title | string | What this execution does |
| objective | text | Full objective for the agent |
| project_slug | string | Project scope |
| intent_slug | string | Intent folder reference (nullable for non-intent work) |
| mode | enum | research / outline / implement / review / refactor / harvest |
| status | enum | See lifecycle below |
| requires_approval | boolean | If true, waits for user approval before dispatch |
| proposal_id | FK → Proposal | Origin (nullable — manual/pipe/responsibility executions have no proposal) |
| task_id | FK → Task | Associated task (nullable) |
| session_id | FK → ClaudeSession | Linked CLI session (nullable) |
| brain_dump_item_id | string | Origin brain dump item (nullable) |
| result_path | string | Filesystem path to result artifact |
| git_diff | text | Captured diff after execution |
| metadata | map | Extension point for governance, cost, routing data |
| completed_at | datetime | When execution finished |

**Status lifecycle (LOCKED):**
```
created → proposed → awaiting_approval → approved → delegated → running → completed
                                                                       → failed
                                                                       → cancelled
                                              → harvesting → completed
```

**Every input source collapses into Execution:**

| Input Source | How It Becomes an Execution |
|-------------|----------------------------|
| Proposal (approved) | `Executions.on_proposal_approved/1` — already wired |
| Manual dispatch (API) | `POST /api/executions` with `requires_approval: false` |
| Responsibility (due task) | Pipe: `responsibilities:task_generated` → `create_execution` action |
| Pipe action | New action `"executions:create"` in Pipes.Registry |
| Brain dump (direct) | Future: Clusterer creates intent + proposal, or user manually approves |
| Scheduled (cron) | Seed fires → Generator → Proposal → approve → Execution |

## 2. The Execution-Centered Spine

### Principle

> **If EMA does meaningful work, an Execution row exists for it.**

This is non-negotiable. The Execution table is EMA's memory of what it has done, is doing, and plans to do.

### What bypasses proposals

These inputs create Executions directly (no proposal required):

1. **Manual dispatch** — User creates execution via API with `requires_approval: false`
2. **Pipe-triggered execution** — A pipe action creates an execution (e.g., on task completion, run a review)
3. **Responsibility-generated execution** — Due responsibility → task → execution (via pipe)

Everything else goes through proposals:
- Brain dump items → (cluster into intent) → seed → proposal → execution
- Scheduled seeds → proposal → execution
- Git/session/vault events → harvester → seed → proposal → execution

### How Responsibilities map to Executions

```
Responsibility (cadence fires)
  → Responsibility.Scheduler generates Task (source_type: "responsibility")
  → Pipe trigger "tasks:created" fires
  → Pipe action "executions:create" creates Execution with:
      mode: depends on responsibility role
      requires_approval: true (default)
      task_id: the generated task
```

### How Pipes map to Executions

New stock action added to Registry:

```elixir
{"executions:create", "Create an execution from pipe event",
  fn payload ->
    Ema.Executions.create(%{
      title: payload["title"] || "Pipe-triggered execution",
      objective: payload["objective"],
      project_slug: payload["project_slug"],
      mode: payload["mode"] || "implement",
      requires_approval: Map.get(payload, "requires_approval", true),
      metadata: %{source: "pipe", trigger: payload["_trigger_pattern"]}
    })
  end}
```

### How Agents map to Executions

Agents do not freelance. Every agent invocation is an AgentSession belonging to an Execution. The `AgentWorker.dispatch_to_domain/3` path must create or link to an Execution. Casual agent chat (webchat, Discord) is the only exception — these are conversations, not executions.

## 3. The Core Loop

### NOW (implementable immediately)

```
Proposal (status: "queued")
  → User approves (or auto-approval policy fires)
  → Executions.on_proposal_approved/1         [EXISTS]
  → Execution created (status: "approved")     [EXISTS]
  → Broadcast {:dispatch, execution}           [EXISTS]
  → Dispatcher receives                        [EXISTS]
  → Dispatcher builds delegation packet        [EXISTS]
  → Dispatcher calls Runner.run/2              [DECISION: Runner, not Bridge]
  → Result received
  → Execution → status: "completed"            [EXISTS]
  → IntentFolder.write_result + append_log     [EXISTS]
  → Event recorded                             [EXISTS]
  → PubSub broadcast "execution:completed"     [EXISTS]
```

**What's actually missing to make this work end-to-end:**
1. Dispatcher must handle Runner failures gracefully (retry once, then fail)
2. `approve_and_dispatch/1` must be called when `requires_approval: false` — currently only dispatches if not requiring approval, but the flag defaults to `true`
3. Auto-approval needs a trigger (governance policy or pipe)

### LATER (additive, no refactor)

```
brain_dump → [CLUSTER] → intent folder created → seed generated
  → ProposalEngine generates proposal
  → [GOVERNANCE CHECK] → auto-approve or queue for review
  → Execution created
  → [COMMITTEE INTERCEPT] → routing/mode selection
  → Dispatcher runs
  → Result harvested
  → [MEMORY WRITE] → vault note, intent log, reflexion lesson
  → [NEXT PROPOSAL] → ReflectionLoop suggests follow-up seed
```

**Extension points (marked with brackets above):**

| Hook | Where it plugs in | What it touches |
|------|-------------------|-----------------|
| CLUSTER | New module `BrainDump.Clusterer` | Reads unprocessed items, writes intent folders, creates seeds. Zero changes to Execution. |
| GOVERNANCE CHECK | `Executions.create/1` and `approve_and_dispatch/1` | Reads policies, writes Event with actor_kind "governance". May block dispatch. |
| COMMITTEE INTERCEPT | `Dispatcher` before building packet | Calls committee modules for mode/routing decisions. Writes Event. |
| MEMORY WRITE | `on_execution_completed/2` | Already calls IntentFolder + ReflectionLoop. Vault write is additive. |
| NEXT PROPOSAL | `ReflectionLoop.reflect_async/1` | Already exists. Can create follow-up seeds. |

**None of these change the Execution lifecycle or Dispatcher flow.** They are pre-hooks and post-hooks on existing functions.

## 4. Governance Layer (Minimal, Real)

### A. Policy Source

**DB table: `governance_policies`** (new, simple)

```elixir
schema "governance_policies" do
  field :name, :string          # "auto_approve_xs", "agent_cost_limit"
  field :rule_type, :string     # "auto_approve" | "dispatch_constraint" | "cost_limit"
  field :conditions, :map       # %{"scope" => "xs", "confidence_gte" => 0.7}
  field :action, :map           # %{"approve" => true} or %{"max_cost_usd" => 5.0}
  field :active, :boolean       # toggle on/off
  field :priority, :integer     # lower = higher priority, for conflict resolution
  timestamps()
end
```

No files. No constitution system. Just rows the user can manage via API.

### B. Enforcement Points

**Two enforcement points only:**

1. **Pre-dispatch gate** in `Executions.approve_and_dispatch/1`:
   ```
   Before dispatching, call Governance.check_dispatch(execution)
   → Returns :ok | {:block, reason}
   → If blocked, transition to "awaiting_approval", record Event
   → If ok, proceed to dispatch
   ```

2. **Auto-approval hook** in `Executions.on_proposal_approved/1`:
   ```
   After creating execution, call Governance.maybe_auto_approve(execution)
   → Checks policies: if proposal.confidence >= threshold AND scope <= limit → auto-approve
   → If auto-approved, skip "awaiting_approval" → go straight to dispatch
   → Record Event with actor_kind "governance"
   ```

**Not an enforcement point:** Result acceptance. Results are always accepted and recorded. Quality evaluation is a follow-up execution (mode: "review"), not a gate.

### C. First Three Policies

**1. Auto-approval for low-risk proposals**
```elixir
%{
  name: "auto_approve_small_confident",
  rule_type: "auto_approve",
  conditions: %{"estimated_scope_in" => ["xs", "s"], "confidence_gte" => 0.7},
  action: %{"auto_approve" => true},
  active: true,
  priority: 10
}
```

**2. Agent cost limit per execution**
```elixir
%{
  name: "cost_ceiling",
  rule_type: "cost_limit",
  conditions: %{},
  action: %{"max_cost_usd" => 10.0, "max_retries" => 1},
  active: true,
  priority: 5
}
```

**3. Scope advisor block (repeated failure prevention)**
```elixir
%{
  name: "scope_advisor_block",
  rule_type: "dispatch_constraint",
  conditions: %{"check_scope_advisor" => true},
  action: %{"block_if_repeated_failure" => true},
  active: true,
  priority: 1
}
```

### D. Implementation

New module: `Ema.Governance` (single file, ~80 lines)

```elixir
defmodule Ema.Governance do
  def check_dispatch(execution) do
    policies = list_active_policies("dispatch_constraint") ++ list_active_policies("cost_limit")
    Enum.reduce_while(policies, :ok, fn policy, :ok ->
      case evaluate(policy, execution) do
        :ok -> {:cont, :ok}
        {:block, reason} -> {:halt, {:block, reason}}
      end
    end)
  end

  def maybe_auto_approve(execution, proposal) do
    policies = list_active_policies("auto_approve")
    Enum.any?(policies, &matches?(&1, execution, proposal))
  end
end
```

## 5. Intent Layer Placement

### Where brain dump items live

**DB: `brain_dump_items` table (exists).** They stay there. Unprocessed items have `processed: false`.

### Where clustering occurs

**New module (future): `Ema.BrainDump.Clusterer`**

Not built now. When built, it:
1. Reads unprocessed brain_dump_items with embeddings
2. Groups by cosine similarity (threshold: 0.75)
3. For each cluster: creates intent folder via `IntentFolder.create/3`
4. Creates a Seed pointing to that intent
5. Marks items as `processed: true, action: "processing"`

**This module has zero dependencies on the core loop.** It reads BrainDump items and writes Seeds + intent folders. The core loop picks up from Seeds via ProposalEngine.

### How proposals generate from intent

Already works:
1. Seed has `prompt_template` with `{intent}` placeholder
2. Generator reads intent folder contents via `IntentFolder` module
3. Generator interpolates into prompt, calls Runner
4. Proposal created with `intent_slug` in metadata

**What's missing:** The Generator doesn't currently read intent folder contents. It reads project context via ContextManager. Adding intent folder reading to Generator is a single function call addition.

### Guarantee: Intent does NOT block execution

Intent is a filesystem-resident knowledge layer. If no intent exists:
- `execution.intent_slug` is nil
- `IntentFolder.write_result` silently no-ops
- The execution still completes normally

Intent is always additive. The core loop works with or without it.

## 6. Committee Model (Constrained)

### What a "committee" is in code

A committee is a **pure function module** that takes an execution context and returns a decision.

```elixir
defmodule Ema.Committees.Committee do
  @callback deliberate(context :: map()) :: {:ok, decision :: map()} | {:abstain, reason :: String.t()}
end

# Example implementation
defmodule Ema.Committees.ModeSelector do
  @behaviour Ema.Committees.Committee

  def deliberate(%{execution: exec, intent_history: history, project_context: ctx}) do
    # Analyze intent execution log to pick optimal next mode
    # Returns {:ok, %{recommended_mode: "review", confidence: 0.8, reasoning: "..."}}
  end
end
```

### Where committees intercept (MAX 2 points)

**Point 1: Pre-dispatch mode selection** (in Dispatcher, before packet building)

```elixir
# In Dispatcher.dispatch/1, after building initial packet:
case Committees.consult(:mode_selection, %{execution: execution, intent_history: history}) do
  {:ok, %{recommended_mode: mode}} -> %{packet | mode: mode}
  {:abstain, _} -> packet  # use execution's original mode
end
```

**Point 2: Post-completion next-action recommendation** (in on_execution_completed)

```elixir
# In on_execution_completed/2, after recording result:
case Committees.consult(:next_action, %{execution: execution, result: result}) do
  {:ok, %{create_seed: seed_attrs}} -> Proposals.create_seed(seed_attrs)
  {:ok, %{create_execution: exec_attrs}} -> Executions.create(exec_attrs)
  {:abstain, _} -> :noop
end
```

### Committees.consult/2

```elixir
defmodule Ema.Committees do
  @registry %{
    mode_selection: [Ema.Committees.ModeSelector],
    next_action: [Ema.Committees.ReflectionAdvisor]
  }

  def consult(decision_point, context) do
    members = Map.get(@registry, decision_point, [])
    votes = Enum.map(members, & &1.deliberate(context))
    # For now: first non-abstain vote wins (single member per committee)
    Enum.find(votes, {:abstain, "no votes"}, &match?({:ok, _}, &1))
  end
end
```

**Not built now.** The two hook points in Dispatcher and on_execution_completed are the only integration. When committee modules exist, they plug in. Until then, both return `{:abstain, _}` and the system behaves as-is.

## 7. Hard Technical Decisions

### A. Execution storage

**DB** for execution metadata, status, relationships. Already exists.  
**Filesystem** for artifacts: `~/.local/share/ema/results/{execution_id}/result.md` (already implemented in `on_execution_completed`).  
**Intent folders** for accumulated knowledge: `.superman/intents/{slug}/` (already implemented).

No change needed.

### B. Dispatch path: Runner

**Runner. Not Bridge.**

The Bridge GenServer is unreliable — it silently falls back to Runner anyway. Maintaining two code paths adds complexity for zero benefit today.

**Decision:**
- `Dispatcher` calls `Runner.run/2` directly
- `ExecutionBridge.dispatch_async/2` wraps Runner in Task.async for non-blocking dispatch
- Bridge module stays in codebase but is not in the critical path
- When Bridge is proven reliable, swap `Runner.run` → `Bridge.run` in one line in Dispatcher

**Cost tracking** (currently in Bridge): Move to Dispatcher post-execution. Parse Runner output for token counts, write to execution.metadata.

### C. Result handling: Async

**Async dispatch, async result handling.**

The Dispatcher already broadcasts `{:dispatch, execution}` via PubSub. The dispatch itself runs in a Task under ExecutionBridge's DynamicSupervisor (max 10 concurrent). Results flow back via:

1. Runner.run returns synchronously within the Task
2. Task calls `Executions.on_execution_completed/2`
3. on_execution_completed writes result, patches intent, records event, broadcasts

The user sees execution status changes via WebSocket channel "execution:{id}".

### D. Streaming: No for core loop, yes for UI

The core loop does not stream. Runner is called, returns a result, result is stored.

For UI feedback during long executions: the ExecutionBridge.Worker already broadcasts progress events to `"executions:{id}:stream"`. This is cosmetic, not structural.

### E. Vault role: Secondary artifact store

**Primary memory:** Intent folders (`.superman/intents/`)  
**Secondary storage:** Vault (`~/.local/share/ema/vault/`)

The vault receives:
- SystemBrain state files (already working)
- Session capture summaries (already working via SessionCapture)
- Execution result copies (additive — write to vault after writing to intent folder)

The vault is NOT the source of truth for execution results. Intent folders are.

## 9. Single Vertical Slice

**Flow: User approves a queued proposal → artifact written + execution completed**

### Step-by-step with exact modules

```
1. User clicks "Approve" in frontend ProposalsApp
   → POST /api/proposals/:id/approve
   → EmaWeb.ProposalController.approve/2

2. Controller calls Ema.Proposals.approve_proposal(id)
   → Sets proposal.status = "approved"
   → Broadcasts "proposal_approved" to "proposals:events"
   → Broadcasts "proposals:approved" to EventBus (pipe trigger)
   → Calls Ema.Executions.on_proposal_approved(proposal.id)

3. Executions.on_proposal_approved/1
   → Creates Execution:
     %{
       title: proposal.title,
       objective: proposal.body,
       project_slug: project.slug,
       intent_slug: IntentFolder.slugify(proposal.title),
       mode: "implement",
       status: "approved",
       requires_approval: false,  # already user-approved
       proposal_id: proposal.id
     }
   → Records Event: %{type: "created", actor_kind: "system"}
   → Records Event: %{type: "approved", actor_kind: "user"}
   → Broadcasts {:dispatch, execution} to "executions:dispatch"

4. Ema.Executions.Dispatcher receives {:dispatch, execution}
   → Calls Governance.check_dispatch(execution) → :ok (stub)
   → Calls Committees.consult(:mode_selection, ctx) → {:abstain, _} (stub)
   → Builds delegation packet:
     %{
       execution_id: "exec_abc123",
       project_slug: "proslync",
       intent_slug: "add-oauth-flow",
       agent_role: "implementer",
       objective: "...",
       mode: "implement",
       success_criteria: ["Implementation compiles", "Tests pass"],
       read_files: [".superman/intents/add-oauth-flow/intent.md"],
       write_files: [".superman/intents/add-oauth-flow/result.md"]
     }
   → Fetches project context via ContextBuilder
   → Fetches Superman context via Superman.context_for(project_slug)
   → Fetches reflexion prefix via ReflexionInjector
   → Creates AgentSession:
     %{execution_id: exec.id, agent_role: "implementer", status: "pending", prompt_sent: full_prompt}
   → Transitions execution: status → "delegated"
   → Transitions execution: status → "running"
   → Updates AgentSession: status → "running"
   → Calls Runner.run(full_prompt, model: "sonnet", timeout: 300_000)

5. Runner.run/2 returns {:ok, result_json}
   → Dispatcher extracts result_summary from result_json
   → Updates AgentSession: status → "completed", result_summary: result_summary
   → Calls Executions.on_execution_completed(execution.id, result_summary)

6. Executions.on_execution_completed/2
   → Transitions execution: status → "completed"
   → Writes result artifact: ~/.local/share/ema/results/{exec_id}/result.md
   → Writes intent result: IntentFolder.write_result(project_path, intent_slug, result_summary)
   → Appends intent log: IntentFolder.append_log(project_path, intent_slug, exec_id, "implement", result_summary)
   → Records Event: %{type: "completed", actor_kind: "system", payload: %{result_summary: "..."}}
   → Broadcasts {"execution:completed", %{execution: execution}} to "executions"
   → Calls Committees.consult(:next_action, ctx) → {:abstain, _} (stub)
   → Calls ReflectionLoop.reflect_async(execution) (if module exists)

7. PubSub consumers react:
   → SessionCapture writes vault summary
   → Pipes trigger on "executions:completed" (if pipe configured)
   → Frontend execution channel pushes status update to UI
```

| DB: execution_events | 4 rows: created, approved, status_changed×2, completed |

## Appendix: What This Architecture Guarantees

1. **Every meaningful action is an Execution** — auditable, traceable, with events
2. **The core loop works with Runner alone** — no Bridge dependency
3. **Intent is additive** — the loop works without intent folders; they accumulate value over time
4. **Governance is two hooks** — pre-dispatch and auto-approval, nothing more
5. **Committees are pure functions** — they advise, they don't control flow
6. **Future systems plug in at marked points** — no refactor needed
7. **Dead code is identified and scheduled for removal** — no pretending

This is the spine. Build on it.
