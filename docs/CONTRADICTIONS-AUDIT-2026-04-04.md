# EMA Contradictions Report — 2026-04-04

Generated by: systematic audit of CLAUDE.md claims, router.ex, user_socket.ex, channel files, store files, application.ex, and key module files.

---

## Critical (blocks functionality)

**1. Bridge is NOT started by default — but code everywhere calls Bridge.run()**

- **Claimed:** CLAUDE.md "Next Phase" section describes Bridge as the multi-backend AI system. Multiple production modules call `Ema.Claude.Bridge.run/2` directly: `voice_controller.ex` (lines 87, 107), `focus/summary.ex` (line 37), `pipes/actions/claude_action.ex` (line 150), `agent_worker.ex` (line 181), `agent_memory.ex` (line 93), `evolution/instruction_parser.ex` (line 16), `voice_core.ex` (lines 193, 222).
- **Reality:** `config/config.exs` sets `ai_backend: :runner`. `application.ex` only starts `Ema.Claude.BridgeSupervisor` when `Application.get_env(:ema, :ai_backend) == :bridge` (line 103). With `:runner` as default, `BridgeSupervisor` is never started. However, `Bridge.run/2` has a fallback: if `GenServer.whereis(__MODULE__)` returns nil, it falls back to `Runner.run/2` (bridge.ex `do_run` function). So calls to `Bridge.run` don't crash — but they silently bypass the entire Bridge machinery (smart routing, cost tracking, governance, circuit breakers) and fall through to the Runner. The Bridge is described as an active system but is never actually started.
- **Evidence:** `daemon/config/config.exs:ai_backend: :runner`; `daemon/lib/ema/application.ex:maybe_start_bridge/0`; `daemon/lib/ema/claude/bridge.ex:do_run` fallback
- **Impact:** All Bridge features (multi-account rotation, provider failover, cost tracking, quality gates, governance audit logging) silently don't apply on every AI call.

**2. Seven frontend stores join channels that don't exist on the backend**

Frontend stores call `joinChannel("X")` where no channel module registers that topic:

| Store | joinChannel call | No matching channel join |
|-------|-----------------|--------------------------|
| `token-monitor-store.ts` | `"intelligence:tokens"` | No channel file has `def join("intelligence:tokens"...)` |
| `vm-health-store.ts` | `"intelligence:vm"` | No channel file has `def join("intelligence:vm"...)` |
| `jarvis-store.ts` | `"jarvis:lobby"` | No `jarvis_channel.ex` file exists |
| `prompt-workshop-store.ts` | `"prompts:lobby"` | No channel file has `def join("prompts:lobby"...)` |
| `decision-store.ts`/`decision-log-store.ts` | `"decisions:lobby"` | No channel with that join |
| `pipeline-store.ts` | `"pipeline:lobby"` | No channel file has `def join("pipeline:lobby"...)` |
| `project-graph-store.ts` | `"project_graph:lobby"` | No channel file has `def join("project_graph:lobby"...)` |

- **Reality:** These 7+ channel topics are registered in `user_socket.ex` via wildcard patterns, but none of the actual channel `def join` clauses handle these specific topic strings. Clients attempting to join these channels will receive Phoenix join errors.
- **Evidence:** `app/src/stores/token-monitor-store.ts:101`, `vm-health-store.ts:56`, `jarvis-store.ts:142`, `prompt-workshop-store.ts:68`; cross-checked against all `def join` clauses in `daemon/lib/ema_web/channels/`.

**3. Harvesters described as "designed but not implemented" — but vault/usage/brain_dump harvesters are missing module files entirely**

- **Claimed:** CLAUDE.md "Stubs / Needs Work" says: "Harvesters (Git, Session, Vault, Usage, BrainDump) — designed but not implemented." `harvesters/run.ex` line 22 lists `@valid_harvesters ~w(git session vault usage brain_dump)`. `Harvesters.Supervisor` starts `GitHarvester` and `SessionHarvester`.
- **Reality:** Only `git_harvester.ex` and `session_harvester.ex` exist in `daemon/lib/ema/harvesters/`. There are no `vault_harvester.ex`, `usage_harvester.ex`, or `brain_dump_harvester.ex` files. The `@valid_harvesters` list in `run.ex` declares all five as valid, but three are missing module files entirely. Any code that attempts to start or call `Ema.Harvesters.VaultHarvester`, `Ema.Harvesters.UsageHarvester`, or `Ema.Harvesters.BrainDumpHarvester` will fail with `UndefinedFunctionError`.
- **Evidence:** `daemon/lib/ema/harvesters/` directory listing; `run.ex:22`

## Significant (docs misleading, creates confusion)

**5. CLAUDE.md says "13 apps" — the router, stores, and module list indicate ~50 virtual apps**

- **Claimed:** CLAUDE.md Frontend Structure section: "App Components (13 apps)" listing BrainDumpApp through ChannelsApp.
- **Reality:** The router has controllers for Contacts, Finance, Invoices, Routines, Meetings, Clipboard, Tunnels, FileVault, MessageHub, TeamPulse, Goals, Focus, Voice, Superman, Evolution, MetaMind, Organizations, Temporal, Security, VmHealth, Tokens, Intent, Gaps, ProjectGraph, Memory, ExecutionBoard, AiSessions, ClaudeSessions, Git Intelligence, CliManager, OpenClaw — far more than 13.
- **Evidence:** `daemon/lib/ema_web/router.ex` contains routes for ~50 distinct controller modules. The "13 apps" count is severely outdated.

**6. CLAUDE.md says "approve → Task" — reality is approve → Execution → AI agent dispatch**

- **Claimed:** CLAUDE.md: "User actions: approve (→ Task), redirect (→ 3 new seeds), kill (→ KillMemory)." Implies approve creates a Task.
- **Reality:** `proposals.ex` `approve_proposal/1` calls `Ema.Executions.on_proposal_approved(proposal.id)`. This creates an `Execution` record, transitions it to "approved", then dispatches to an AI agent via `executions:dispatch` PubSub → `Dispatcher`. The approved proposal doesn't directly create a Task — it creates an Execution that is delegated to Claude. The path is: approval → Execution creation → AI agent dispatch → agent presumably creates task as output. The CLAUDE.md "approve → Task" description hides a significant layer of complexity.
- **Evidence:** `daemon/lib/ema/proposals/proposals.ex:54`; `daemon/lib/ema/executions/executions.ex:89-120`

**7. The 2026-03-29 design spec is for a completely different app ("place-native")**

- **Claimed:** `docs/superpowers/specs/2026-03-29-ema-design.md` is listed as a key EMA spec file.
- **Reality:** It is titled "place-native — Design Spec", targets "KDE Neon (Ubuntu 24.04), Plasma 6.6, Wayland", and references data stored at `~/.local/share/place-native/` with a `place.db` database. This is an entirely different predecessor project. EMA uses `~/.local/share/ema/ema.db`. Any developer reading this spec for EMA context will be severely misled.
- **Evidence:** `docs/superpowers/specs/2026-03-29-ema-design.md` header vs `daemon/config/runtime.exs:54`

**8. Agent "tool execution" described as working but is 1-of-many implemented**

- **Claimed:** CLAUDE.md "Working" section includes "Agent system with per-agent supervision, memory compression, and webchat bridge." `AgentWorker` is described as "handling messages, Claude CLI calls, tool execution."
- **Reality:** CLAUDE.md "Stubs / Needs Work" admits: "Agent tool execution — only `brain_dump:create_item` implemented." The description implies a full tool execution system; the reality is a single stub tool.
- **Evidence:** CLAUDE.md stubs section.

**9. Vault path resolution uses four different methods — compile_env vs get_env conflict**

- `context_builder.ex:14` uses `Application.compile_env(:ema, :vault_path, ...)` — baked in at compile time, ignores runtime config changes.
- `wiki_sync.ex:172` uses `Application.get_env(:ema, :vault_path, ...)` — reads at runtime.
- `vault_seeder.ex:101` uses `System.get_env("EMA_VAULT_PATH") || Path.expand(...)` — env var based.
- `second_brain.ex:449` hardcodes `Path.expand("~/.local/share/ema/vault")` directly.
- **Impact:** If someone overrides vault path in runtime config (e.g., `config :ema, vault_path: "/custom/path"`), `context_builder.ex` will still use the compiled default, while `wiki_sync.ex` will use the runtime value. Context injection and wiki sync diverge.
- **Evidence:** All four files listed above.

## Verified Correct (spot-checks that passed)

- **Route → Controller file existence:** All controller files referenced in `router.ex` exist in `daemon/lib/ema_web/controllers/`. No missing controller modules found.
- **Channel registration → file existence:** All channel modules in `user_socket.ex` have corresponding `.ex` files. No registered-but-missing channel modules.
- **Vault default path consistency:** The default path `~/.local/share/ema/vault/` is consistent across all four resolution methods (methods differ but defaults match).
- **Bridge fallback safety:** `Bridge.run/2` gracefully falls back to `Runner.run/2` when Bridge GenServer isn't started. No crash risk from Bridge.run calls with `:runner` backend.
- **Proposal approval → Execution dispatch chain wired end-to-end:** `proposals.ex:approve_proposal` → `Ema.Executions.on_proposal_approved` → create Execution → `transition("approved")` → `dispatch_if_ready` → `Phoenix.PubSub.broadcast("executions:dispatch", {:dispatch, execution})` → `Dispatcher` handles `{:dispatch, execution}`. Full chain exists.
- **Harvesters Supervisor structure correct:** `Ema.Harvesters.Supervisor` exists with proper OTP structure supervising the two implemented harvesters.
- **ProposalEngine pipeline stages all exist:** `scheduler.ex`, `generator.ex`, `refiner.ex`, `debater.ex`, `tagger.ex`, `combiner.ex`, `kill_memory.ex` all present.
- **Claude Sessions system matches CLAUDE.md description:** `ClaudeSessions.Supervisor` starts conditionally (default true). All described sub-modules (`SessionWatcher`, `SessionMonitor`, `SessionParser`, `SessionLinker`) exist.
- **Most channel/store pairs match:** The majority of stores use correct channel topic strings that match actual `def join` clauses. The failures are isolated to the 7 listed above.

## Recommendations (top 5 fixes by impact)

**1. Fix the 7 dead WebSocket channel topics** *(High impact — frontend apps silently fail to connect)*

For each of `intelligence:tokens`, `intelligence:vm`, `jarvis:lobby`, `prompts:lobby`, `decisions:lobby`, `pipeline:lobby`, `project_graph:lobby`:
- Either create the missing channel module with a `def join` handler
- Or update the frontend store to use an existing channel topic
- Ensure wildcard registration in `user_socket.ex` covers the topic if needed

**2. Activate Bridge in production config or document `:runner` as intentional** *(High impact — silent feature degradation on every AI call)*

- Option A: Set `config :ema, :ai_backend, :bridge` in `config/config.exs` to activate Bridge and all its features
- Option B: If `:runner` is intentional, audit and remove Bridge calls from `voice_controller.ex`, `pipes/actions/claude_action.ex`, `agent_worker.ex`, etc. — replace with `Runner.run` or `AI.run` for consistency
- Document in CLAUDE.md which modules use Bridge vs Runner and why

**3. Implement or remove the 3 missing harvester modules** *(Medium impact — `@valid_harvesters` references undefined modules)*

- Create `vault_harvester.ex`, `usage_harvester.ex`, `brain_dump_harvester.ex` as stub modules using `Ema.Harvesters.Base` behaviour (implement `harvest/1` with a `{:ok, %{}}` stub)
- Add them to `Harvesters.Supervisor` children list when ready
- Or remove them from `@valid_harvesters` to avoid misleading validation

**4. Standardize vault path resolution to a single config function** *(Medium impact — config overrides silently broken)*

- Replace `Application.compile_env` in `context_builder.ex:14` with `Application.get_env` (allows runtime override)
- Create `Ema.Config.vault_path/0` that all modules call instead of each resolving independently
- Update `second_brain.ex` hardcoded path and `vault_seeder.ex` env-var logic to use the shared function

**5. Update CLAUDE.md to reflect current scale and correct misleading claims** *(Medium impact — onboarding confusion)*

- Fix store count: "15 Zustand stores" → "67+ stores"
- Fix app count: "13 apps" → accurate figure (~50 virtual apps/controllers)
- Add note that `docs/superpowers/specs/2026-03-29-ema-design.md` is a historical artifact for the predecessor project "place-native", not EMA
- Expand "approve → Task" description to explain the full Execution/Dispatcher pipeline
- Clarify that Bridge is opt-in via `ai_backend: :bridge` config and what features are lost with the default `:runner`
